# AI 協作紀錄

依開發階段記錄與 AI（Claude Code）的協作過程。

| Prompt | AI 回覆摘要 | 採用/修改 | 修改原因 | 驗證方式 |
|---|---|---|---|---|
| 執行階段 1 | 依 `dev_prompt.md` 階段 1 規格，建立 `backend/` FastAPI 專案：`app/database.py`（SQLAlchemy engine/session）、`app/models.py`（`Event` ORM，含 `ImageType` enum）、`app/schemas.py`（Pydantic `EventCreate`/`EventUpdate`/`EventOut`/`ImageParams`，含 date/time 格式驗證與 image_type/image_source 一致性檢查）、`app/crud.py`（CRUD 函式）、`app/main.py`（6 支 API：`POST/GET /events`、`PUT/DELETE /events/{id}`、`POST /upload`、靜態掛載 `/uploads`，並設定 CORS 開放 `localhost:5173`）。 | 採用 | 完全對應 CLAUDE.md 的資料模型與 API 規格；額外加上 image_type 為 none 時清空 image_source 的一致性驗證，避免髒資料 | 建立 Python 3.11 虛擬環境並安裝 `requirements.txt`，啟動 `uvicorn` 後以 Python `urllib` 腳本與 `curl` 逐一測試：建立事件（含中文標題、image_params 巢狀物件）、依月份查詢、更新、刪除、刪除後查詢確認排除、上傳圖片（含拒絕非圖片檔案）、靜態圖片服務（200/404）。所有情境回應狀態碼與內容皆符合預期，測試後已清除暫存的 `events.db` 與上傳檔案 |
| 執行階段 2（前端月曆） | 開工前先依 dev_prompt.md 指出兩處規格模糊點並詢問使用者：①階段2 modal 的「新增/編輯」要做完整表單還是整個留到階段3；②月曆縮圖本階段是否就要用 Canvas 依 image_params 渲染。取得回覆後，以 Vite 建立 `frontend/`，實作 `src/api.js`（封裝 events/upload API）、`src/utils/imageRender.js`（唯一的 Canvas 渲染函式，供三處渲染共用）、`src/utils/dateUtils.js`（月曆日期計算）、元件 `CanvasThumbnail`/`DayCell`/`Calendar`/`DayModal`/`EventForm`，並整合進 `App.jsx`；`index.css` 手刻無框架樣式。功能涵蓋年月切換、跳轉指定年月、日期格事件顯示與 +N 溢出、modal 內新增/編輯/刪除/預覽，圖片僅支援上傳或外部網址（二選一），縮圖與預覽皆由 Canvas 依 image_params 渲染。 | 採用 | 兩個問題使用者皆選擇建議方案：①完整表單但圖片欄位只做上傳/URL、進階編輯留階段3；②現在就建立共用 Canvas 縮圖渲染元件，避免階段3再回頭改三處渲染邏輯。另外做了兩個實作面判斷：Image 載入不設定 `crossOrigin`（避免外部圖片網址未回應 CORS 標頭時整張圖載入失敗，本專案只做畫面渲染不需讀取像素資料）；`offsetX`/`offsetY` 定義為相對畫布尺寸的比例而非像素，讓同一組參數在縮圖與大預覽等不同尺寸的畫布上偏移比例一致——此為階段2內部渲染設計決策，屬純技術細節、不影響已確認的產品範圍，故未另外詢問，若階段3打造編輯器時發現不合適會再調整 | 用 `npx playwright install chromium` 安裝無頭瀏覽器，背景啟動 backend（port 8000）與 `npm run dev`（port 5173），寫 Playwright 腳本自動操作：切換上/下月、跳轉到 2026-12、點日期格開 modal、新增含上傳圖片的事件、確認縮圖與 modal 預覽以 Canvas 正確渲染（含透明背景棋盤格）、編輯標題並確認更新、刪除事件並確認列表回到空狀態、全程 `console --errors` 為空陣列；另外用 API 直接建立同一天 5 筆事件，截圖確認日期格顯示 3 筆 + `+2` 溢出、modal 可捲動列出全部 5 筆。驗證完成後已刪除所有測試事件並關閉 dev server |
| 執行階段 3（事件表單圖片編輯器） | 開工前先指出 dev_prompt.md 未定義「裁切/顯示區域」的具體互動方式（拖曳裁切框 vs. 數值 slider），詢問使用者後採用拖曳式裁切框。實作三個新檔案：`src/components/CropBoxEditor.jsx`（顯示原圖 + 可拖曳移動/可拖曳四角縮放的裁切框，寫入 `crop.{x,y,width,height}`，角落縮放採「對角錨點固定」演算法，避免因 React 重繪造成的 closure 過期問題）、`src/components/ImageEditor.jsx`（整合裁切框、套用完整 image_params 的即時預覽 canvas〔在其上拖曳寫入 offsetX/offsetY〕、縮放/旋轉〔slider + 90° 步進按鈕〕/透明度/亮度/對比 slider，所有變動都立即呼叫共用的 `drawImageWithParams` 重繪，不做防抖）；修改 `EventForm.jsx`：新增 `imageParams` state 取代原本寫死的 `DEFAULT_IMAGE_PARAMS`、新增 `normalizeImageParams()` 補齊舊資料缺漏欄位、換新圖片時重置參數為預設值、移除舊有的靜態 `CanvasThumbnail` 預覽改由 `ImageEditor` 內建即時預覽取代；`index.css` 新增對應樣式。 | 採用 | 裁切互動方式使用者明確選擇「拖曳式裁切框」（相對於純數值 slider 或兩者都做），故不自行假設實作方式；縮放/旋轉/位移/裁切/透明度/亮度對比皆延續階段2已定義的座標單位（crop 相對原圖比例、offset 相對畫布比例、scale 以裁切後圖片 contain-fit 為基準），全部透過既有的 `drawImageWithParams` 共用渲染，未另開新的渲染路徑 | 用 Playwright 產生一張四色象限＋圓點的測試圖片（紅/綠/藍/黃四象限，中心黑點，用於肉眼辨識裁切/旋轉/位移效果），背景啟動 backend（8000）與 frontend（5173），自動化腳本依序驗證：①初始預覽四象限顏色與位置正確；②旋轉 slider 拖到 90° 後用截圖比對（左上變藍/右上變紅/右下變綠/左下變黃，確認為順時針旋轉，符合程式碼「順時針為正」的設計）；③+90°/-90° 按鈕正確寫入 rotation；④透明度 slider 立即降低 canvas alpha 值；⑤拖曳裁切框右下角控制點，用截圖確認裁切框確實縮小、框外出現半透明遮罩；⑥在預覽 canvas 拖曳圖片，確認可調整位移；⑦縮放/亮度/對比 slider 皆可即時生效；⑧儲存後 reload 整個頁面、重新開啟同一事件的編輯表單，確認 scale/brightness/crop 等參數皆從後端正確還原（並直接呼叫 `GET /events` API 核對 image_params JSON 內容），月曆縮圖刷新後也維持正確渲染。全程 console errors 為 0。過程中一次寫的測試腳本因為在多個 UI 操作後才量測裁切框座標、量測時機過舊（layout 已因先前操作改變），導致裁切拖曳誤判為未生效，用截圖比對後定位是量測時機問題而非產品邏輯錯誤，修正腳本後（改為互動前先 `scrollIntoViewIfNeeded()` 再即時量測座標）確認裁切功能本身正常。驗證完成後已刪除所有測試事件、清除 `backend/uploads/` 內本次上傳的測試圖檔，並關閉 dev server |
| 執行階段 4（儲存還原驗證） | 開工前先詢問使用者「手動測試清單要放在哪裡」（獨立檔案 vs. 併入尚未建立的 README.md），使用者選擇「先建立 README.md，清單為其中一節」。據此建立 `README.md`（含專案總覽、架構圖、資料結構、image_params 設計理念、前後端資料流、啟動說明、API 文件、完成/未完成清單、設計取捨、手動測試清單）。清單涵蓋 5 大情境：A. 多事件×多圖片來源混合還原、B. 各 image_params 欄位邊界值還原（含 rotation 負值/270°、crop 極小、opacity=0 等）、C. 新增→多次編輯→刷新→再編輯不互相覆蓋、D. 刪除後刷新不殘留、E. 月曆縮圖/modal 預覽/表單編輯器三處渲染一致性。清單寫完後實際撰寫 Playwright 腳本執行全部 5 類情境（共 21 項檢查），而非只靠程式碼推論。 | 採用 | README 章節結構依使用者選擇的方案建立；清單覆蓋範圍依使用者需求逐條列出的 5 大類情境設計，未自行縮減範圍 | 啟動 backend（8000）/frontend（5173），在暫存目錄安裝 Playwright + chromium + pngjs，寫自動化腳本：用四象限＋中心黑點測試圖驗證，A 類在同一天建立無圖片/上傳圖片/外部網址三事件並核對還原；B 類建立 9 筆單一邊界值事件，讀取編輯器滑桿與文字顯示數值核對還原（`scale`/`rotation`/`opacity`/`brightness`/`contrast` 讀文字，`offsetX/Y`/`crop` 因編輯器無數值顯示改用「截圖後於 Node 端以 pngjs 解析像素」比對顏色分佈佐證，因為圖片來源跨來源會使 canvas 遭污染、`getImageData` 依專案設計會拋 SecurityError，改用瀏覽器截圖不受此限制）；C 類對同一事件連續編輯兩次（先改 rotation 再改 brightness，中間不刷新）後刷新驗證兩次修改都保留且不互相覆蓋，並直接呼叫 `GET /events` API 核對 DB 內容；D 類刪除事件後確認 modal/月曆/API 皆不殘留；E 類對同一筆 `rotation:90` 事件比對月曆縮圖/modal 列表縮圖/lightbox 預覽/表單編輯器即時預覽四處的四角顏色分佈，結果完全一致。過程中修正了 3 個測試腳本本身的問題（非產品問題）：①外部圖片來源原本用 httpbin.org/image/png 在瀏覽器內載入失敗（跨網域圖片服務行為問題，非本專案程式碼），改用穩定可載入的 `w3.org` 圖示網址；②同一天事件數超過 `DayCell` 顯示上限 3 筆時，直接用事件標題找日期格會找不到（該標題被摺進 "+N" 溢出未顯示），改用「該天內一定會顯示的錨點標題」開啟日期格，modal 內清單仍能列出全部事件；③預覽按鈕文字實際是 `aria-label` 而非可見文字，selector 需求要對應調整。最終共 21 項檢查點全數通過（PASS），全程 console errors 為 0。測試中另外發現一個邊界案例（非本次修復範圍，先記錄於 README）：`rotation` 還原值若超出編輯器滑桿可視範圍（-180°~180°，如資料為 270°），滑桿手把的 DOM 顯示會被瀏覽器 native 行為鉗制在 180°，但文字標籤與實際存檔數值都正確無誤，此情境在正常 UI 操作流程下不會發生（滑桿與 90° 按鈕都會把角度正規化到 -180°~180°），只有外部直接寫入超出範圍值時才會出現，是否處理待使用者確認。驗證完成後已刪除所有測試事件（API 確認 DB 清空）、清除本次與先前階段殘留的所有測試上傳圖檔，並關閉 dev server |
